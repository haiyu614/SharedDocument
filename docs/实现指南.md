# 协同文档系统 - 分步实现指南

## 项目概述

本项目旨在开发一款功能完善的共享编辑文档系统，支持多人实时协作编辑、版本管理、权限控制等功能。

---

## 总体实现顺序

按照依赖关系，建议按以下顺序实现：

1. 环境搭建与数据库初始化 → 基础环境
2. 用户认证系统 → 所有功能的基础
3. 文档基础CRUD → 核心功能
4. 版本管理 → 基于文档CRUD
5. 实时协作编辑 → 最复杂的功能

### 为什么是这个顺序？

#### 依赖关系说明

第一步：环境搭建（必须先做）
- 没有数据库，无法存储任何数据
- 没有开发环境，无法运行代码
- 这是所有功能的基础，必须最先完成

第二步：用户认证（必须先做）
- 后续所有功能都需要知道"谁在使用"
- 权限控制依赖用户身份
- 文档创建需要知道"谁创建的"
- 版本管理需要记录"谁保存的"

第三步：文档CRUD（核心功能）
- 版本管理需要先有文档才能保存版本
- 实时协作需要先有文档才能编辑
- 这是系统的核心功能，必须先完成

第四步：版本管理（基于CRUD）
- 需要先有文档才能保存版本
- 相对独立，可以先完成
- 实时协作时也会用到版本保存功能

第五步：实时协作（最复杂）
- 需要前面的功能都完成
- 技术难度最高，需要WebSocket和OT算法
- 需要文档、用户、版本等所有基础功能

#### 顺序调整建议


并行开发顺序
第一步完成后，第二步和第三步可以并行开发：
- 后端开发A：专注第二步（认证接口）
- 后端开发B：专注第三步（文档接口）
- 前端开发A：同时做第二步（登录页面）和第三步（文档列表）
适合场景：团队协作、时间紧迫、多人开发

最小可行顺序（MVP演示）
如果时间紧迫，可以简化第四步和第五步：
- 第一步、第二步、第三步：必须完整实现
- 第四步：先实现基础保存功能，对比功能可以延后
- 第五步：先实现简单同步，完整的OT算法可以延后
适合场景：快速演示、功能验证、时间紧迫

## 第一步：环境搭建与数据库初始化

### 目标
- 配置开发环境
- 创建数据库表结构
- 初始化基础数据（角色）

### 后端任务清单

#### 1. 创建数据库
在 MySQL 中执行以下 SQL 语句创建数据库：

CREATE DATABASE collab_doc CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

#### 2. 配置环境变量
创建 backend/.env 文件，设置以下变量：
- DATABASE_URL：数据库连接地址
- SECRET_KEY：JWT 密钥
- CORS_ALLOW_ORIGINS：允许的前端地址

#### 3. 初始化 Alembic（数据库迁移工具）
在 backend 目录下执行：
alembic init alembic

#### 4. 创建迁移文件
创建 alembic/versions/001_initial.py 文件，定义所有表结构：
- roles 表：角色信息
- users 表：用户信息
- documents 表：文档信息
- document_versions 表：文档版本
- edit_logs 表：编辑日志

#### 5. 执行数据库迁移
执行命令：alembic upgrade head

#### 6. 初始化角色数据
执行脚本：python scripts/init_data.py
创建三个默认角色：admin（管理员）、editor（编辑者）、viewer（查看者）

### 前端任务清单

#### 1. 安装依赖
在 frontend 目录下执行：npm install

#### 2. 配置环境变量
创建 frontend/.env 文件，设置：
- VITE_API_BASE_URL：后端 API 地址
- VITE_WS_BASE_URL：WebSocket 地址

#### 3. 验证启动
执行命令：npm run dev
访问 http://localhost:5173 确认前端可以正常启动

## 第二步：用户认证系统

### 目标
- 用户注册功能
- 用户登录功能（JWT Token）
- 获取当前用户信息
- 前端路由守卫

### 后端要做什么

#### 已完成的基础代码
- app/api/routes/auth.py：路由已定义
- app/services/auth_service.py：服务层已实现
- app/core/security.py：JWT 和密码加密已实现

#### 需要测试和验证
1. 启动后端服务：uvicorn app.main:app --reload
2. 测试注册接口：POST /api/auth/register
3. 测试登录接口：POST /api/auth/login
4. 测试获取用户信息：GET /api/auth/me（需要 Token）

### 前端要做什么

#### 1. 完善登录页面（frontend/src/pages/LoginPage.vue）
- 添加表单验证（用户名、密码必填）
- 调用 AuthService.login() 接口
- 保存 Token 到 localStorage
- 登录成功后跳转到文档列表

#### 2. 实现路由守卫（frontend/src/router/index.ts）
- 检查用户是否登录
- 未登录用户访问受保护页面时，自动跳转到登录页
- 已登录用户访问登录页时，自动跳转到首页

#### 3. 完善认证 Store（frontend/src/store/index.ts）
- login()：登录并保存 token
- logout()：登出并清除 token
- hydrateFromStorage()：从 localStorage 恢复状态
- getCurrentUser()：获取当前用户信息

#### 4. 测试登录流程
- 注册新用户
- 使用用户名密码登录
- 刷新页面，Token 仍然有效
- 登出后无法访问受保护页面

## 第三步：文档基础CRUD

### 目标
- 创建文档
- 获取文档列表
- 获取单个文档详情
- 更新文档（标题）
- 删除文档（软删除）

### 后端要做什么

#### 1. 完善文档服务层（backend/app/services/document_service.py）
实现以下方法：
- create_document(owner_id, title, initial_content)：创建新文档，同时创建第一个版本
- list_documents(user_id, role_name)：根据用户权限返回文档列表
- get_document(document_id, user_id)：获取文档详情，检查权限
- update_document(document_id, user_id, title)：更新文档标题
- delete_document(document_id, user_id)：软删除文档

#### 2. 完善文档路由（backend/app/api/routes/documents.py）
实现以下接口：
- POST /api/documents：创建文档
- GET /api/documents：获取文档列表
- GET /api/documents/{id}：获取文档详情
- PUT /api/documents/{id}：更新文档
- DELETE /api/documents/{id}：删除文档

#### 3. 实现权限检查
- admin：可以查看/编辑所有文档
- editor：可以查看/编辑自己创建的文档
- viewer：只能查看文档

### 前端要做什么

#### 1. 完善文档列表（frontend/src/pages/EditorPage.vue）
- 调用 DocumentService.listDocuments() 获取列表
- 在侧边栏显示文档列表
- 点击文档项，加载文档内容到编辑器

#### 2. 实现创建文档功能
- 点击"新建文档"按钮
- 弹出输入框，输入文档标题
- 调用 DocumentService.createDocument() 接口
- 创建成功后自动选中新文档

#### 3. 实现文档选择功能
- 点击侧边栏文档项
- 调用 DocumentService.getDocument(id) 获取详情
- 将文档内容加载到 Quill 编辑器
- 显示当前文档标题

#### 4. 测试文档CRUD
- 创建文档
- 查看文档列表
- 选择文档，查看内容
- 编辑文档标题
- 删除文档

## 第四步：版本管理

### 目标
- 每次保存生成新版本
- 查看版本列表
- 查看历史版本内容
- 版本对比（diff）
- 回滚到指定版本

### 后端要做什么

#### 1. 完善版本服务层（backend/app/services/version_service.py）
实现以下方法：
- save_version(document_id, user_id, content)：保存当前内容为新版本
- list_versions(document_id)：获取文档的所有版本
- get_version(version_id)：获取指定版本内容
- compare_versions(document_id, from_version, to_version)：对比两个版本的差异
- restore_version(document_id, version_id, user_id)：回滚到指定版本

#### 2. 完善版本路由（backend/app/api/routes/versions.py）
实现以下接口：
- POST /api/documents/{id}/save：保存版本
- GET /api/documents/{id}/versions：版本列表
- GET /api/documents/{id}/versions/{version_id}：获取版本
- GET /api/documents/{id}/compare：对比版本
- POST /api/documents/{id}/restore/{version_id}：回滚版本

#### 3. 实现版本对比算法
- 使用 Python difflib 库
- 生成文本差异（增加/删除/修改）

### 前端要做什么

#### 1. 实现保存版本功能（frontend/src/pages/EditorPage.vue）
- 点击工具栏"保存"按钮
- 获取 Quill 编辑器内容
- 调用 DocumentService.saveVersion() 接口
- 显示保存成功提示

#### 2. 实现版本列表页面（frontend/src/pages/VersionsPage.vue）
- 调用 DocumentService.listVersions() 获取版本列表
- 显示版本列表（版本号、创建时间、创建者）
- 点击版本项，查看该版本内容

#### 3. 实现版本对比功能
- 选择两个版本进行对比
- 调用 DocumentService.compareVersions() 接口
- 使用 jsdiff 库高亮显示差异
- 显示增加/删除/修改的内容

#### 4. 实现版本回滚功能
- 在版本列表中点击"回滚"按钮
- 弹出确认对话框
- 调用 DocumentService.restoreVersion() 接口
- 回滚成功后刷新文档内容

#### 5. 测试版本管理
- 创建文档，编辑内容
- 多次保存版本
- 查看版本列表
- 对比不同版本
- 回滚到旧版本

## 第五步：实时协作编辑

### 目标
- WebSocket 连接管理
- 实时同步编辑操作
- 显示其他用户光标
- 显示在线用户列表

### 后端要做什么

#### 1. 实现 WebSocket 端点（backend/app/api/routes/collab.py）
实现 WebSocket 连接处理：
- 验证 token，获取用户信息
- 接受连接
- 将连接加入文档会话
- 广播用户加入消息
- 监听消息（编辑操作、光标更新）
- 处理操作（OT transform）
- 广播操作到其他客户端
- 处理断开连接

#### 2. 实现协同服务（backend/app/services/collab_service.py）
创建 CollaborationSession 类：
- add_client()：添加客户端
- remove_client()：移除客户端
- apply_operation()：应用编辑操作（OT transform）
- broadcast()：广播消息到所有客户端

#### 3. 实现 OT 算法
- 简化版：基于文本位置的转换
- 或使用现成库（如 ot-text）

### 前端要做什么

#### 1. 实现 WebSocket 连接（frontend/src/pages/EditorPage.vue）
- 打开文档时建立 WebSocket 连接
- 连接参数：document_id, token
- 处理连接成功/失败
- 处理断开重连

#### 2. 实现编辑操作同步
- 监听 Quill 编辑器的 text-change 事件
- 将编辑操作转换为 OT 格式
- 通过 WebSocket 发送到服务器
- 接收服务器广播的操作
- 应用到本地编辑器（避免循环触发）

#### 3. 实现光标同步
- 监听编辑器光标变化
- 发送 cursor_update 消息
- 接收其他用户的光标位置
- 在编辑器中显示其他用户光标（不同颜色）

#### 4. 实现在线用户显示
- 接收 user_joined / user_left 消息
- 更新协作者列表
- 在界面上显示在线用户头像

#### 5. 测试实时协作
- 打开两个浏览器窗口
- 都登录不同账号
- 打开同一个文档
- 在一个窗口编辑，观察另一个窗口是否同步
- 观察光标是否同步显示

## 开发建议

### 1. 严格按照顺序实现（重要）
- 必须完成第一步（环境搭建）才能开始后续步骤
- 必须完成第二步（用户认证）才能实现需要权限的功能
- 必须完成第三步（文档CRUD）才能实现版本管理和实时协作
- 不要跳步骤，否则会遇到依赖问题

### 2. 先完成后端接口，再实现前端
- 使用 Postman 测试所有 API
- 确保接口返回正确的数据格式
- 后端接口测试通过后，前端再调用

### 3. 逐步测试，不要一次性实现所有功能
- 每完成一个小功能就测试
- 确保当前功能正常再继续
- 每完成一个步骤，进行完整的验收测试

### 4. 团队协作时的顺序建议
- 第一步：所有人一起完成（环境搭建）
- 第二步和第三步：可以并行开发（不同人员负责不同模块）
- 第四步和第五步：建议按顺序完成（依赖关系较强）

### 5. 使用 Git 版本控制
- 每个步骤完成后提交一次
- 提交信息清晰标注完成的步骤
- 方便回滚和协作

### 6. 遇到问题及时记录
- 记录错误信息和解决方案
- 方便后续参考
- 团队共享问题和解决方案

## 总结

本实现指南按照从简单到复杂的顺序，将项目分为五个步骤逐步实现。每个步骤都有明确的目标、任务清单和验收标准。建议严格按照顺序实现，每完成一步就进行测试，确保功能正常后再继续下一步。

最终目标是实现一个功能完善的多人实时协作文档编辑系统，支持用户认证、文档管理、版本控制和实时协作等核心功能。
